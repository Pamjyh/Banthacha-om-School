## 0. Purpose & elevator pitch

ระบบออมทรัพย์นักเรียนโรงเรียนบ้านท่าชะอมต้องส่ง "ทะเบียนการรับฝากเงินประจำวัน" + "ใบปะหน้านำฝาก ธ.ก.ส." ให้ธนาคารทุกวันที่มีการฝาก ปัจจุบัน Pam (แอดมิน) สร้างเอกสารนี้ด้วยมือใน Google Sheets ทุกครั้ง (copy จากไฟล์เดือนก่อน แก้ตัวเลข) ซึ่งพบว่ามีจุดผิดพลาดสะสมจากการทำมือ (ดูหัวข้อ "ปัญหาที่พบจากไฟล์ตัวอย่างจริง" ด้านล่าง)

ฟีเจอร์นี้คือปุ่มในหน้า **admin** ของเว็บ (ระบบออมทรัพย์) ที่กดแล้วสร้างเอกสารทั้ง 2 หน้า (ทะเบียนรับฝาก + ใบปะหน้า) ให้อัตโนมัติจากข้อมูลธุรกรรมจริงในระบบ สำหรับวันที่เลือก แทนที่การพิมพ์มือทั้งหมด **ยกเว้นส่วนนับธนบัตร/เหรียญจริง** ซึ่งเป็นข้อมูลทางกายภาพที่ระบบรู้ไม่ได้ ต้อง Pam กรอกเองหลังนับเงินสดจริง

## 1. Core model / approach

**Non-negotiable invariants:**
- ต้องส่งธนาคารได้จริง → ฟอร์แมต/คำ/โครงสร้าง ต้องตรงกับไฟล์ตัวอย่างที่ธนาคารเคยรับมาแล้ว (อ้างอิงไฟล์จริง: "ทะเบียนเงินสดฝากประจำวันโรงเรียนบ้านท่าชะอม กรกฎาคม 2569.xlsx" ใน Google Drive ของ Pam, fileId `1CHJBZkncgoTXKbgEMO6NcgXITJTZ88yz`)
- ยอดรวมใน Sheet1 (ทะเบียน) ต้อง**เท่ากับ**ยอดที่ Sheet2 อ้างอิงมาโชว์เทียบเสมอ (คำนวณจากข้อมูลเดียวกัน ไม่พิมพ์แยก)
- ห้ามแก้ไข/ลบข้อมูลธุรกรรมหรือนักเรียนใดๆ ระหว่างสร้างฟอร์ม (read-only จาก sheet "ธุรกรรม" และ "นักเรียน")
- ต่อยอดจากฟังก์ชัน `exportMonthly()` ที่มีอยู่แล้วใน `Code.gs` (ระบบออมทรัพย์) — ใช้ pattern การสร้าง sheet ใหม่/จัดฟอร์แมตแบบเดียวกัน ไม่ต้องคิด styling ใหม่จากศูนย์

**ขอบเขต (IN scope):**
- สร้าง Sheet1 (ทะเบียนรับฝากประจำวัน) + Sheet2 (ใบปะหน้านำฝาก) ในสเปรดชีตเดียวกับข้อมูลหลัก (SHEET_ID เดิม)
- เลือกวันที่ได้ (ไม่ใช่แค่วันนี้) ผ่าน date picker ฝั่งแอดมิน
- รวมทุกชั้นเป็นทะเบียนเดียว เรียงตามลำดับที่เกิดธุรกรรม (ไม่แยกชีตต่อชั้น)
- ช่องกรอกธนบัตร/เหรียญเปล่าให้ Pam พิมพ์เอง พร้อมสูตรคำนวณอัตโนมัติในสเปรดชีต

**ขอบเขต (OUT / deferred — ไม่ทำรอบนี้):**
- ไม่ทำ auto-download `.xlsx` ตรง (ดูเหตุผลใน Decision Log #7) — Pam เปิด sheet แล้วกด File > Download เองถ้าต้องการไฟล์จริง
- ไม่ auto-increment เลขที่เอกสาร (พิมพ์เอง)
- ไม่ผูก "ผู้รับเงิน" กับระบบ auth ที่มีอยู่ (พิมพ์เองตอนสร้าง)
- ไม่รวมนักเรียนที่ยังไม่ได้ฝากวันนั้นในทะเบียน (ต่างจากไฟล์เก่าที่เคยรวม)

## 2. ปัญหาที่พบจากไฟล์ตัวอย่างจริง (ทำไมฟีเจอร์นี้ต้องออกแบบตามนี้)

ตรวจไฟล์ตัวอย่างจริง (127 รายการ, กรกฎาคม 2568) พบ 3 จุด ซึ่งเป็นที่มาของการตัดสินใจออกแบบในไฟล์นี้:

1. **เลขลำดับที่ (ลำดับที่) ซ้ำกัน** — เพราะไฟล์เดิม layout เป็น 2 คอลัมน์ซ้าย-ขวาต่อหน้า (ลำดับ 1-22 ซ้าย, 23-45 ขวา) แล้วมีแถวเกินที่ทำให้ทั้งสองฝั่งใช้เลข "23" ซ้ำกันกับนักเรียนคนละคน → **แก้โดยออกแบบให้เป็นรายชื่อเดียวต่อเนื่อง ไม่มี 2 คอลัมน์** (Decision Log #4)
2. **คอลัมน์ "จำนวน" กับ "ยอดรวม" ไม่ตรงกันในแถวหนึ่ง** (0 vs 20 บาท) — เพราะพิมพ์ 2 ช่องแยกกันด้วยมือ พิมพ์ผิดจุดหนึ่งแล้วอีกจุดไม่ sync → **แก้โดยให้ระบบคำนวณ 2 คอลัมน์นี้จากค่าเดียวกันเสมอ (auto-fill)** (Decision Log #3)
3. **"จำนวนผู้ฝากทั้งหมด 155 ราย" ใน Sheet2 ไม่ตรงกับจำนวนแถวจริงในทะเบียน (127 แถว)** — ไม่ทราบที่มาของเลข 155 แน่ชัด (อาจเป็นยอดรวมนักเรียนทั้งโรงเรียน ไม่ใช่ยอดผู้ฝากวันนั้น) → **แก้โดยกำหนดว่า "จำนวนผู้ฝากทั้งหมด" = จำนวนแถวในทะเบียนที่สร้างเสมอ (คำนวณจากข้อมูลเดียวกัน ไม่มีทางไม่ตรงกัน)** (Decision Log #6)

## 3. Data sources & schema (จาก Code.gs ที่มีอยู่แล้ว — อ่านก่อนสร้าง)

### 3.1 Sheet "นักเรียน" (ใน spreadsheet `SHEET_ID`)
Header แถวแรก มีคอลัมน์ (ชื่อ header อาจมีหลายชื่อที่ยอมรับ — ใช้ dynamic lookup แบบเดียวกับฟังก์ชันอื่นๆ ใน Code.gs เสมอ ห้าม hardcode ตำแหน่งคอลัมน์):

| Field (internal) | Header names ที่ยอมรับ (findCol) |
|---|---|
| id | `id` |
| name | `ชื่อ-สกุล`, `ชื่อ สกุล`, `ชื่อ` |
| grade | `ชั้นปัจจุบัน`, `ชั้น` |
| bankAccount | `เลขบัญชี_ธกส`, `เลขบัญชีธกส`, `เลขบัญชี` |
| rollNo | `เลขที่` |
| status | `สถานะ` |

### 3.2 Sheet "ธุรกรรม"
Header แถวแรก: `id, นักเรียน_id, ชื่อ, ชั้น, ประเภท, จำนวนเงิน, ปีการศึกษา, วันที่`
- `id` = `T<epoch ms>` (string)
- `ประเภท` = `'ฝาก'` หรือ `'ถอน'` (string, exact match)
- `วันที่` = **⚠️ แก้ไข 2026-07-07 หลังตรวจ Code.gs จริง**: string รูปแบบไทย `"D MMM YYYY HH:MM"` **มีเวลาต่อท้ายเสมอ** เช่น `"7 ก.ค. 2569 14:30"` — สร้างจาก `thaiDate(new Date())` ซึ่งจริงๆ append `HH:MM` เสมอ (ดูฟังก์ชัน `thaiDate(d)` บรรทัด ~1175 ใน Code.gs: `return d.getDate()+' '+THAI_MONTHS[d.getMonth()]+' '+y+' '+hh+':'+mm;`) — สมมติฐานเดิมในร่างแรกของไฟล์นี้ว่า "ไม่มีเวลา" ผิด แก้ไขแล้ว

### 3.3 การกรองธุรกรรมตามวันที่ (สำคัญ — ใช้ prefix match ไม่ใช่ exact equality)
เพราะ `วันที่` มีเวลาต่อท้ายเสมอ (ต่างกันทุกแถว) จะ exact-match ทั้ง string ไม่ได้ ต้องแปลงวันที่ที่แอดมินเลือกเป็น string เฉพาะส่วนวันที่ `"D MMM YYYY"` (ไม่มีเวลา, เดือนใช้ `THAI_MONTHS`/`MONTHS_SHORT`, ปี พ.ศ. = ค.ศ.+543) แล้ว filter ด้วย **prefix match**: `String(row[idxDate]).indexOf(dateStr) === 0` — รูปแบบเดียวกับที่ `getHistory()` ใช้อยู่แล้ว (`r.date.indexOf(p.date) === 0`) และเดียวกับที่ `loadTodaySummary()` ใน index.html ส่ง `todayStr` (ไม่มีเวลา) ไปให้ `getHistory({action:'getHistory', date: todayStr})` — ห้ามใช้ `===` เทียบตรงๆ เด็ดขาด จะได้ 0 แถวเสมอ

## 4. Trigger / Backend action spec

**ฟังก์ชันใหม่ใน Code.gs**: `generateDepositRegistry(p)`

Action string ใหม่ใน `handleRequest()`: `case 'generateDepositRegistry': result = generateDepositRegistry(params); break;`

**Params (จาก frontend):**
| param | type | ตัวอย่าง | หมายเหตุ |
|---|---|---|---|
| `password` | string | — | admin gate: `if (p.password !== ADMIN_PASSWORD) return {ok:false, error:'เฉพาะผู้ดูแล'}` (pattern เดียวกับ `exportMonthly`) |
| `date` | string | `"7 ก.ค. 2569"` | รูปแบบ Thai date string ตรงกับที่เก็บใน sheet ธุรกรรม (ดู 3.3) |
| `docNo` | string | `"003"` | เลขที่เอกสาร พิมพ์เอง ไม่ validate format |
| `receiverName` | string | `"พศุตม์"` | ชื่อผู้รับเงิน พิมพ์เอง |

**ขั้นตอนภายในฟังก์ชัน:**
1. เช็ค admin password
2. อ่านทุก transaction ที่ `ประเภท==='ฝาก'` และ `วันที่ === p.date` (exact match) จาก sheet ธุรกรรม
3. ถ้าไม่มีธุรกรรมเลย → return `{ok:false, error:'ไม่พบธุรกรรมฝากในวันที่ '+p.date}`
4. เรียงตาม `id` (เวลาที่บันทึกจริง — ทำให้เลขลำดับที่สะท้อนลำดับการฝากจริง) — **ตัดสินใจ**: ใช้ลำดับเวลาการฝาก ไม่ใช่เรียงตามเลขที่นักเรียน (เพราะทะเบียนธนาคารบันทึกตามลำดับที่มารับฝากจริง)
5. สำหรับแต่ละ transaction, join กับ sheet นักเรียนด้วย `นักเรียน_id` เพื่อดึง `bankAccount` (เลขบัญชี ธกส.) — ถ้าไม่มี (ว่าง) ให้ปล่อยว่างไว้ในทะเบียน (Decision Log #10) ไม่ตัดแถวออก
6. สร้าง Sheet1 ตามสเปคข้อ 5
7. สร้าง Sheet2 ตามสเปคข้อ 6 (อ้างอิงยอดรวมจาก Sheet1)
8. return `{ok:true, sheet1Name, sheet2Name, url, totalAmount, depositorCount}` (url ชี้ไป Sheet1 โดย fragment `#gid=`, pattern เดียวกับ `exportMonthly`)

## 5. Sheet1 spec — ทะเบียนการรับฝากเงินประจำวัน

ชื่อ sheet: `'ทะเบียนฝาก_' + <D-M-YYYY ของ p.date, เช่น 7-7-2569>` — ถ้ามีชื่อซ้ำ ให้ลบของเดิมทิ้งก่อนสร้างใหม่ (pattern เดียวกับ `exportMonthly`: `if (out) ss.deleteSheet(out);`)

Layout แถวต่อแถว (single column ต่อเนื่อง — **ไม่มี 2 คอลัมน์ซ้าย-ขวา** ตาม Decision Log #4):

| แถว | เนื้อหา |
|---|---|
| 1 | `เลขที่เอกสาร <docNo>` (คอลัมน์ A, bold, พื้นเหลือง — เหมือนไฟล์ตัวอย่าง) |
| 2 | ชื่อโรงเรียน `SCHOOL_NAME` (merge, center, bold, size 14) |
| 3 | `ทะเบียนการรับฝากเงินประจำวันที่ <p.date>` (merge, center, bold) |
| 4 (header) | `ลำดับที่ \| เลขที่บัญชี \| ชื่อบัญชี \| จำนวน \| ยอดรวม \| ผู้รับเงิน \| หมายเหตุ` (bold, พื้นเขียว `#92D050` เหมือนไฟล์ตัวอย่าง, wrap, vertical middle) |
| 5..N | 1 แถวต่อ 1 transaction: `ลำดับที่` = running index เริ่ม 1 ไม่ซ้ำ, `เลขที่บัญชี` = bankAccount (ว่างได้), `ชื่อบัญชี` = ชื่อนักเรียน, `จำนวน` = amount, `ยอดรวม` = amount (**ค่าเดียวกับจำนวนเสมอ** — เขียนจากตัวแปรเดียวกัน ไม่ใช่สูตรแยก, กัน Decision Log #3), `ผู้รับเงิน` = `p.receiverName` ทุกแถว, `หมายเหตุ` = ว่าง |
| N+1 | แถวรวม: `รวมทั้งหมด` + `จำนวนแถว` cell (label ชัดว่าใช้เป็น "จำนวนผู้ฝากทั้งหมด" ต่อใน Sheet2) + ผลรวม `จำนวน`/`ยอดรวม` ทุกแถว (bold, พื้นเขียวเข้ม/ดำเหมือนแถวรวมใน `exportMonthly`) |

Number format คอลัมน์จำนวน/ยอดรวม: `#,##0` (pattern เดียวกับ `exportMonthly`)
Border รอบตารางทั้งหมด: `setBorder(true,true,true,true,true,true)`
Column width: ลำดับที่ 55px, เลขที่บัญชี 130px, ชื่อบัญชี 180px, จำนวน/ยอดรวม 80px, ผู้รับเงิน/หมายเหตุ 80px (ค่าเดียวกับ `exportMonthly`)

## 6. Sheet2 spec — ใบปะหน้านำฝาก ธ.ก.ส.

ชื่อ sheet: `'ปะหน้าฝาก_' + <D-M-YYYY>` — ลบของเดิมก่อนถ้าซ้ำชื่อ

| ส่วน | เนื้อหา |
|---|---|
| หัว | `ใบปะหน้านำฝาก ธ.ก.ส. <SCHOOL_NAME>` / `ใบตรวจนับเงินสดคงเหลือ` / `วันที่ <p.date>` |
| ตารางนับธนบัตร/เหรียญ | คอลัมน์: `รายการตรวจนับเงินสด \| มูลค่าต่อหน่วย \| จำนวนหน่วย (Pam กรอกเอง — ช่องว่าง) \| จำนวนเงิน (สูตร = มูลค่า×จำนวนหน่วย)` แถวธนบัตร: 1000, 500, 100, 50, 20, 10 — แถวเหรียญ: 10, 5, 2, 1, 0.50, 0.25 |
| ยอดรวม (1) | `ยอดรวมเงินสดที่ตรวจนับได้จริง (ธนบัตร+เหรียญ) (1)` = **สูตร SUM** ของคอลัมน์ "จำนวนเงิน" ทั้งหมด (คำนวณสดเมื่อ Pam กรอกจำนวนหน่วย) |
| ยอดรวม (2) | `ยอดรวมเงินสดตามทะเบียนเงินสด (2)` = ค่าคงที่จากยอดรวม Sheet1 (เขียนตอนสร้าง ไม่ใช่สูตรอ้างอิงข้ามชีต — เพื่อกันปัญหาถ้า Pam ลบ/ย้าย Sheet1 ทีหลัง) |
| ผลต่าง | `ผลต่าง (1)-(2)` = **สูตรลบ** อ้างอิงเซลล์ (1) กับ (2) ในชีตเดียวกัน — recalculate สดทุกครั้งที่ Pam แก้จำนวนหน่วย พร้อมข้อความ `*** ผลต่างต้องเท่ากับศูนย์ (0)` |
| จำนวนผู้ฝาก | `จำนวนผู้ฝากทั้งหมด <depositorCount> ราย` — ค่าคงที่ = จำนวนแถวจริงใน Sheet1 (แก้ปัญหา #6 ใน Decision Log) |
| ลายเซ็น | 3 บรรทัด: กรรมการผู้รักษาเงินสดโรงเรียนธนาคาร (1./2./3. เว้นว่างให้เซ็นมือ) + พนักงาน ธ.ก.ส. (ผู้จัดการ/ผู้ช่วยผู้จัดการ, วันที่/เวลา) + ครูผู้รับมอบเงิน (ตำแหน่ง, วันที่) — เว้นว่างทั้งหมดให้เซ็นมือ (เหมือนไฟล์ตัวอย่าง) |
| ท้าย | `**** เอกสารนำส่ง ธ.ก.ส.` |

## 7. Frontend (หน้า admin, index.html)

เพิ่ม section ใหม่ใต้ tab admin ที่มีอยู่แล้ว (`ov`/`tx`/`pr`/`gd`) — ชื่อ tab แนะนำ: `🏦 ทะเบียนฝากธนาคาร`

องค์ประกอบ:
- Date picker (native `<input type="date">`) — เลือกวันไทยตามปกติ ต้องแปลงเป็น Thai date string ก่อนส่ง (ใช้ helper แปลงแบบเดียวกับที่ `loadTodaySummary()`/`todayStr` ใช้ — เดือนต้องเป็น `THAI_MONTHS`/`MONTHS_SHORT` array ไม่ใช่ `toLocaleDateString`)
- ช่องกรอก `เลขที่เอกสาร` (text)
- ช่องกรอก `ผู้รับเงิน` (text)
- ปุ่ม `📄 สร้างทะเบียน`
- หลังกดสำเร็จ: แสดงลิงก์เปิด Google Sheet ที่สร้าง (เหมือนที่ `exportMonthly`/`exportTerm` ทำอยู่แล้วในหน้านี้) + สรุปสั้นๆ (จำนวนผู้ฝาก, ยอดรวม)
- error handling: ถ้า `!r.ok` โชว์ `r.error` ตรงๆ (เช่น "ไม่พบธุรกรรมฝากในวันที่ ...")

## 8. Edge cases & validation

| กรณี | พฤติกรรมที่ตัดสินใจ |
|---|---|
| ไม่มีธุรกรรมฝากเลยในวันที่เลือก | return error ชัดเจน ไม่สร้าง sheet เปล่า |
| นักเรียนฝากแต่ไม่มีเลขบัญชี ธกส. | รวมในทะเบียน เว้นคอลัมน์เลขบัญชีว่าง (Decision Log #10) |
| นักเรียนถูกลบไปแล้วหลังฝาก (join ไม่เจอ) | แสดงชื่อจาก `ชื่อ` ที่บันทึกไว้ใน transaction เอง (ธุรกรรมมี snapshot ชื่อ+ชั้นอยู่แล้วตอนบันทึก ไม่ต้องพึ่ง join 100%) — เลขบัญชีเว้นว่าง |
| ชื่อ sheet ซ้ำ (สร้างทะเบียนวันเดียวกันซ้ำ) | ลบของเดิมทิ้งก่อนสร้างใหม่ (เหมือน `exportMonthly`) — ไม่เตือนก่อนลบ (ตาม pattern เดิมของระบบ) |
| มีการ "ถอน" เงินในวันเดียวกัน | **ไม่รวมในทะเบียนนี้** — ทะเบียนนี้คือ "รับฝาก" เท่านั้น (`ประเภท==='ฝาก'` เท่านั้น ตรงตามชื่อฟอร์ม) |
| Pam กรอกจำนวนหน่วยธนบัตร/เหรียญเป็นค่าติดลบหรือไม่ใช่ตัวเลข | ไม่ validate ฝั่ง Code.gs (เป็น cell ธรรมดาใน Google Sheet ที่ Pam แก้เอง) — สูตร SUM จะคำนวณตามที่กรอกจริง Pam เห็นผลต่างไม่เท่า 0 เองถ้าพิมพ์ผิด |

## 9. Reused conventions จาก exportMonthly (ต้องอ่าน Code.gs ประกอบก่อนสร้างจริง)

- `styleHeader(sheet, cols)` — ใช้จัด header row เดิม
- Pattern ลบ+สร้าง sheet ใหม่ทับชื่อเดิม
- Pattern คืนค่า `url` แบบ `'https://docs.google.com/spreadsheets/d/' + SHEET_ID + '/edit#gid=' + sheet.getSheetId()`
- Number format, border, สีพื้นหลัง ใช้ค่าเดียวกับที่ `exportMonthly` ใช้อยู่แล้ว (ดูโค้ดจริงบรรทัด ~715-922 ของ Code.gs ปัจจุบัน)

## Decision Log

1. เป้าหมาย: แทนที่การสร้างไฟล์นี้ด้วยมือทั้งหมด (Sheet1+Sheet2) เพราะต้องส่งธนาคารจริง ฟอร์แมตต้องตรงของเดิม
2. ขอบเขตข้อมูล: เลือกวันที่ได้ (ไม่ใช่แค่วันนี้) + รวมทุกชั้นในทะเบียนเดียว ไม่แยกต่อชั้น
3. คอลัมน์ "จำนวน"/"ยอดรวม": auto-fill ให้เท่ากันเสมอจากตัวแปรเดียว — กันบั๊กพิมพ์แยก 2 ที่แล้วไม่ตรงกันที่เจอในไฟล์ตัวอย่างจริง (แถว 93)
4. Layout Sheet1: รายชื่อเดียวต่อเนื่อง เลขลำดับไม่ซ้ำ — เลิกรูปแบบ 2 คอลัมน์ซ้าย-ขวาต่อหน้าที่ทำให้เลขลำดับซ้ำในไฟล์ตัวอย่างจริง
5. Sheet2: มีช่องกรอกจำนวนธนบัตร/เหรียญให้ Pam พิมพ์เอง (ระบบรู้ไม่ได้ว่ามีธนบัตรจริงกี่ใบ) + สูตรคำนวณ/เทียบยอดกับ Sheet1 อัตโนมัติในสเปรดชีต
6. รายชื่อในทะเบียน: เฉพาะคนที่ฝากจริงวันนั้น (จำนวน>0) เท่านั้น — "จำนวนผู้ฝากทั้งหมด" นิยามใหม่ = จำนวนแถวในทะเบียนที่สร้างจริง แก้ปัญหาเลข 155 vs 127 ไม่ตรงกันในไฟล์ตัวอย่าง
7. Output: สร้าง sheet ใหม่ในสเปรดชีตเดียวกัน (แบบเดียวกับ `exportMonthly` ที่มีอยู่แล้ว) แล้วส่งลิงก์กลับ — **ไม่ทำ auto-download .xlsx ตรง** เพราะ GAS ไม่มีเครื่องมือสร้าง .xlsx ตรง ต้องผ่านขั้นตอนสร้าง sheet ชั่วคราว + เรียก export + ลบทิ้ง ซึ่งซับซ้อนกว่าและเสี่ยง sheet ชั่วคราวค้างถ้า error กลางทาง — Pam ยืนยันเลือกทางง่ายกว่าหลังรับทราบ trade-off นี้แล้ว
8. ผู้รับเงิน (ชื่อคนรับฝาก): ช่องกรอกเองตอนกดสร้างฟอร์ม ไม่ผูกกับระบบ auth ที่มีอยู่ (ระบบไม่มีชื่อครูแยกรายคน มีแค่รหัสครู/admin รวม)
9. เลขที่เอกสาร: ช่องกรอกเองตอนกดสร้างฟอร์ม ไม่ auto-increment (ไม่ต้องเก็บ state เลขลำดับกลางระบบ)
10. นักเรียนที่ฝากแต่ยังไม่มีเลขบัญชี ธกส. ในระบบ: รวมในทะเบียน แต่เว้นคอลัมน์เลขบัญชีว่างไว้ ไม่ตัดออกจากทะเบียน (เงินเด็กคนนี้ฝากจริง ต้องปรากฏในยอดรวม)
11. เรียงลำดับแถวใน Sheet1: ตามลำดับเวลาที่บันทึกธุรกรรมจริง (`id` = epoch ms) ไม่ใช่เรียงตามเลขที่นักเรียน — สะท้อนลำดับการมารับฝากจริงตามธรรมชาติของทะเบียนธนาคาร
12. ทะเบียนนี้รวมเฉพาะประเภท "ฝาก" เท่านั้น ไม่รวม "ถอน" แม้เกิดวันเดียวกัน

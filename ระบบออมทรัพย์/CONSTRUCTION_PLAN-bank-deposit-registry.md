## วิธีอ่านเอกสารนี้ (สำหรับ Pam)

แต่ละ Stage มี 4 ส่วน:
- 🔨 **สร้าง** — ผมจะทำอะไรในขั้นนี้
- 🧪 **ทดสอบ** — ผมเช็คยังไงว่าถูก (ก่อนให้ Pam ดู)
- 👁️ **Pam ดู** — สิ่งที่ผมจะโชว์ให้ Pam ตรวจ เป็นตัวอย่างจริงจากข้อมูลของ Pam ไม่ใช่ข้อมูลสมมติ
- ✅ **ผ่านเมื่อ** — เงื่อนไขที่ต้องผ่านก่อนไปขั้นถัดไป ถ้าไม่ผ่าน ผมแก้แล้วทดสอบใหม่ก่อน ไม่ข้ามขั้น

หน้าที่ของ Pam ระหว่างสร้าง: ตอบแค่ "ผ่าน" หรือ "ไม่ผ่านเพราะ..." ไม่ต้องอ่านโค้ด

## กฎทองของการสร้างรอบนี้

1. **ข้อมูลต้นทาง (sheet นักเรียน/ธุรกรรม) ห้ามแก้ไข/ลบเด็ดขาด** — อ่านอย่างเดียว
2. **ทุก Stage ต้องผ่านแบบไม่มี error ก่อนไปต่อ** — ไม่ปล่อยผ่านแบบ "น่าจะโอเค"
3. **ตรวจกับข้อมูลจริงของ Pam เสมอ** — ใช้ไฟล์ "ทะเบียนเงินสดฝากประจำวันโรงเรียนบ้านท่าชะอม กรกฎาคม 2569.xlsx" ที่วิเคราะห์ไปแล้วเป็นตัวเทียบ (ground truth ที่ธนาคารรับมาแล้วจริง)
4. **สร้าง engine (คำนวณให้ถูก) ก่อน interface (หน้าตาสวย)** — ถ้าตัวเลขผิด หน้าตาสวยแค่ไหนก็ใช้ไม่ได้ ถ้าเลขถูกแล้วหน้าตาไม่ตรง แก้แค่ styling ไม่ต้องสงสัยตัวเลขอีก
5. **แบ่ง stage เล็กพอที่ถ้าเจอบั๊ก จะรู้ทันทีว่าเกิดจาก stage ไหน**

## แผนที่ Stage

```
PHASE 1 FOUNDATION  — Stage 1-2: routing + auth gate, ดึง/กรองข้อมูลดิบให้ถูกก่อน
PHASE 2 ENGINE       — Stage 3-5: join ข้อมูล, คำนวณยอด, เรียงลำดับ (ยังไม่มีหน้าตา)
PHASE 3 INTERFACE    — Stage 6-8: สร้าง Sheet1, Sheet2, ปุ่ม+ฟอร์มฝั่ง admin
PHASE 4 HARDENING    — Stage 9-11: edge case, เทียบกับไฟล์จริงที่เคยส่งธนาคาร, deploy
```

## Stage 1 — Routing + admin password gate

🔨 **สร้าง**: เพิ่ม action `generateDepositRegistry` ใน `handleRequest()` switch + ฟังก์ชัน `generateDepositRegistry(p)` เปล่าๆ ที่แค่เช็ค `p.password !== ADMIN_PASSWORD` แล้ว return `{ok:true, stub:true}` ถ้ารหัสถูก

🧪 **ทดสอบ**: เรียกด้วยรหัสผิด → ต้องได้ `{ok:false, error:'เฉพาะผู้ดูแล'}` เรียกด้วยรหัสถูก → ต้องได้ `{ok:true, stub:true}`

👁️ **Pam ดู**: ผมจะรายงานผลทดสอบทั้ง 2 เคสเป็นข้อความสั้นๆ ให้ดู (ยังไม่มีอะไรให้กดในเว็บตอนนี้)

✅ **ผ่านเมื่อ**: ทั้ง 2 เคสตรงตามคาด ไม่มี error อื่นแทรก

## Stage 2 — แปลงวันที่ + ดึงธุรกรรม "ฝาก" เฉพาะวันที่เลือก (prefix match)

🔨 **สร้าง**: ฟังก์ชันย่อยแปลงวันที่ input → Thai date string ("D MMM YYYY" ไม่มีเวลา) + logic อ่าน sheet ธุรกรรม กรอง `ประเภท==='ฝาก'` และ `String(วันที่).indexOf(dateStr)===0` (prefix match เพราะ `วันที่` จริงมีเวลาต่อท้ายเสมอ เช่น "7 ก.ค. 2569 14:30" — ตรวจพบระหว่างอ่าน Code.gs จริงก่อนสร้าง แก้ BLUEPRINT แล้ว) — คืนแค่ raw list ของ transaction ที่ตรงเงื่อนไข ยังไม่ join กับนักเรียน

🧪 **ทดสอบ**: รันด้วยวันที่ที่รู้อยู่แล้วว่ามีข้อมูลจริง (จากไฟล์ตัวอย่างที่วิเคราะห์ไปคือ กรกฎาคม 2568 แต่ระบบจริงมีข้อมูลของปีปัจจุบัน 2569 — ใช้วันที่ที่ Pam ยืนยันว่ามีธุรกรรมฝากจริงในระบบตอนนี้) เช็คว่าจำนวนแถวที่ได้ตรงกับที่เปิด Google Sheet ดูเองด้วยตา

👁️ **Pam ดู**: ผมจะโชว์จำนวนธุรกรรมที่ดึงได้ + 3 แถวแรกเป็นตัวอย่าง ("เจอ N รายการฝากวันที่ ... เช่น เด็กชาย/เด็กหญิง ... จำนวน ... บาท") ให้ Pam เทียบกับที่เปิดดูใน Google Sheet เองตรงๆ

✅ **ผ่านเมื่อ**: จำนวน + ตัวอย่างข้อมูลตรงกับที่ Pam เช็คในชีตจริงด้วยตาเอง 100% (นี่คือจุดเสี่ยงสูงสุดของงานนี้ — ถ้าวันที่ match ผิด ทุกอย่างต่อจากนี้ผิดหมด)

## Stage 3 — Join กับข้อมูลนักเรียน (เลขบัญชี ธกส.)

🔨 **สร้าง**: ต่อจาก Stage 2 — join แต่ละ transaction กับ sheet นักเรียนด้วย `นักเรียน_id` เพื่อดึง `เลขบัญชี_ธกส` ถ้าหาไม่เจอ (นักเรียนถูกลบ/ไม่มีเลขบัญชี) ให้เว้นว่างแทนที่จะ error สร้าง array `rows[]` = `{no, bankAccount, name, amount}` (ยังไม่มี ลำดับที่ จริง แค่โครงสร้าง)

🧪 **ทดสอบ**: เช็คว่าทุกแถวมี `name` (ไม่ว่าง เพราะ transaction เก็บชื่อ snapshot ไว้เองอยู่แล้ว) เช็คว่า `bankAccount` ว่างเฉพาะกรณีที่คาดไว้เท่านั้น (นักเรียนที่ยังไม่กรอกเลขบัญชีจริงๆ)

👁️ **Pam ดู**: โชว์ตาราง preview ทุกแถว (ชื่อ + เลขบัญชี + จำนวน) ให้ Pam ไล่ตาดูว่าชื่อ-เลขบัญชีตรงกับที่รู้จักเด็กแต่ละคนจริงไหม โดยเฉพาะแถวไหนที่เลขบัญชีว่าง ให้ชี้ชัดว่าเป็นใคร

✅ **ผ่านเมื่อ**: Pam ยืนยันว่ารายชื่อ+เลขบัญชีถูกต้องครบ ไม่มีคนแปลกปลอม ไม่มีเลขบัญชีผิดคน

---

## Stage 4 — คำนวณยอดรวม + จำนวนผู้ฝาก

🔨 **สร้าง**: คำนวณ `totalAmount` (ผลรวม amount ทุกแถว) และ `depositorCount` (= จำนวนแถว)

🧪 **ทดสอบ**: รวมเลขด้วยมือจากตัวอย่าง preview ใน Stage 3 เทียบกับค่าที่ฟังก์ชันคำนวณได้ ต้องตรงเป๊ะ

👁️ **Pam ดู**: "ยอดรวมฝากวันนี้ = X บาท จาก Y คน" — ให้ Pam เทียบกับยอดที่รู้อยู่แล้วจากการฝากจริงวันนั้น (เช่น เทียบกับที่นับเงินสดได้จริง ถ้ามี)

✅ **ผ่านเมื่อ**: ยอดตรงกับที่ Pam คาด/นับไว้จริง

---

## Stage 5 — เรียงลำดับตามเวลาฝากจริง

🔨 **สร้าง**: เรียง `rows[]` ตาม `id` ธุรกรรม (epoch ms) จากน้อยไปมาก → ได้ลำดับการมาฝากจริงตามเวลา แล้วใส่เลข `ลำดับที่` 1,2,3... ต่อเนื่องไม่ซ้ำ

🧪 **ทดสอบ**: เช็คว่าเลขลำดับที่ 1 ถึง N ไม่ขาด ไม่ซ้ำ (ต่างจากไฟล์ตัวอย่างเดิมที่เจอเลข 23 ซ้ำ)

👁️ **Pam ดู**: โชว์ 5 แถวแรกกับ 5 แถวสุดท้ายพร้อมเลขลำดับที่ ให้ Pam ดูว่าลำดับเวลาสมเหตุสมผลไหม (เช่น คนแรกที่ฝากตอนเช้า ควรเป็นลำดับ 1)

✅ **ผ่านเมื่อ**: เลขลำดับต่อเนื่อง ไม่ซ้ำ ไม่ขาด และ Pam เห็นว่าลำดับเวลาสมเหตุสมผล

---

## Stage 6 — สร้าง Sheet1 จริง (ทะเบียนรับฝาก)

🔨 **สร้าง**: ใช้ผลจาก Stage 1-5 สร้าง Google Sheet ตามสเปคเต็มใน BLUEPRINT หัวข้อ 5 (หัวเรื่อง, header ตาราง, ข้อมูลทุกแถว, แถวรวม, formatting/border/สี ตาม `exportMonthly` pattern)

🧪 **ทดสอบ**: เปิด sheet ที่สร้างขึ้นเอง เช็คว่าจำนวนแถวตรง ผลรวมท้ายตารางตรงกับ Stage 4 ไม่มีเซลล์ว่างผิดที่ ไม่มี label หลุด/เยื้อง

👁️ **Pam ดู**: ผมจะส่งลิงก์ sheet ที่สร้างขึ้นจริงให้ Pam เปิดดูเอง เทียบหน้าตากับไฟล์ตัวอย่างเก่า (กรกฎาคม 2569) ว่าดูเป็นเอกสารแบบเดียวกันไหม ธนาคารน่าจะรับได้ไหม

✅ **ผ่านเมื่อ**: Pam เปิดดูแล้วบอกว่าหน้าตาโอเค พร้อมส่งธนาคารได้ (หรือชี้จุดที่ต้องแก้ให้เหมือนต้นฉบับมากขึ้น)

---

## Stage 7 — สร้าง Sheet2 จริง (ใบปะหน้านำฝาก)

🔨 **สร้าง**: สร้าง Sheet2 ตามสเปค BLUEPRINT หัวข้อ 6 — ตารางนับธนบัตร/เหรียญ (ช่องกรอกว่าง + สูตรคำนวณ), ยอด (1)/(2)/ผลต่าง เป็นสูตร, "จำนวนผู้ฝากทั้งหมด" ดึงจาก Stage 4, ลายเซ็นว่างไว้ให้เซ็นมือ

🧪 **ทดสอบ**: พิมพ์ตัวเลขทดสอบลงช่องจำนวนหน่วยเอง (เช่น ธนบัตร 1000 x 5 ใบ) เช็คว่าสูตร "จำนวนเงิน" และยอดรวม (1) คำนวณถูกจริงในสเปรดชีต แล้วลบตัวเลขทดสอบออกก่อนส่งให้ Pam ดู

👁️ **Pam ดู**: ส่งลิงก์ Sheet2 ให้ Pam ลองพิมพ์จำนวนธนบัตร/เหรียญจริงที่นับได้วันนั้น (ถ้ามีข้อมูลจริงอยู่แล้ว) แล้วดูว่ายอด (1) กับ (2) ตรงกัน ผลต่างออกมาเป็น 0 ไหม

✅ **ผ่านเมื่อ**: สูตรคำนวณถูกต้อง Pam ทดลองพิมพ์เองแล้วยอดตรงตามที่คาด

---

## Stage 8 — หน้า admin: ปุ่ม + ฟอร์ม

🔨 **สร้าง**: เพิ่ม tab ใหม่ `🏦 ทะเบียนฝากธนาคาร` ในหน้า admin (index.html) ตามสเปค BLUEPRINT หัวข้อ 7 — date picker, ช่องเลขที่เอกสาร, ช่องผู้รับเงิน, ปุ่มสร้าง, แสดงผลลัพธ์+ลิงก์หลังสร้างสำเร็จ, แสดง error ชัดเจนถ้าไม่พบธุรกรรม

🧪 **ทดสอบ**: ทดลองกดปุ่มจริงในเบราว์เซอร์ (เลือกวันที่มีข้อมูล, ไม่มีข้อมูล, ลืมกรอกช่อง) เช็คว่าแต่ละเคสแสดงผล/error ถูกต้อง

👁️ **Pam ดู**: ผมจะบอกขั้นตอนให้ Pam ลองกดเองในเว็บจริง (หลัง deploy แล้ว) เลือกวันที่ที่มีข้อมูลจริง แล้วดูว่าได้ลิงก์ sheet ที่ถูกต้อง

✅ **ผ่านเมื่อ**: Pam กดใช้งานจริงในเว็บได้ครบ flow ไม่มีจุดค้าง/error ที่ไม่คาดคิด

---

## Stage 9 — Edge cases sweep

🔨 **สร้าง**: ทดสอบครบทุกเคสใน BLUEPRINT หัวข้อ 8 (ไม่มีธุรกรรม, ไม่มีเลขบัญชี, นักเรียนถูกลบ, สร้างซ้ำวันเดียวกัน, มีทั้งฝากและถอนวันเดียวกัน)

🧪 **ทดสอบ**: รันแต่ละเคสจริงกับข้อมูลทดสอบ (สร้างธุรกรรมทดสอบชั่วคราวถ้าจำเป็น แล้วลบทิ้งหลังทดสอบ — **ห้ามทิ้งข้อมูลทดสอบปนกับข้อมูลจริง**)

👁️ **Pam ดู**: สรุปผลแต่ละเคสเป็นตารางสั้นๆ ("เคส X → ผลลัพธ์ Y ตรงตามที่ตกลงกันไว้ในข้อ Z ของ BLUEPRINT ไหม")

✅ **ผ่านเมื่อ**: ทุกเคสตรงตามที่ตกลงใน BLUEPRINT ไม่มีเคสไหน error แบบไม่คาดคิด

---

## Stage 10 — เทียบกับไฟล์จริงที่เคยส่งธนาคารแล้ว (hand-verify เต็มรูปแบบ)

🔨 **สร้าง**: ไม่มีโค้ดใหม่ — ทำ end-to-end validation เต็มรูปแบบ: ถ้ามีข้อมูลธุรกรรมของวันที่ตรงกับไฟล์ตัวอย่าง "กรกฎาคม 2569" ที่วิเคราะห์ไปแล้ว (หรือวันไหนก็ตามที่ Pam มีไฟล์ต้นฉบับที่ส่งธนาคารไปแล้วจริง) ให้สร้างทะเบียนจากระบบใหม่แล้วเทียบทีละแถวกับไฟล์ต้นฉบับ

🧪 **ทดสอบ**: เทียบยอดรวม, จำนวนคน, รายชื่อ ระหว่างไฟล์เก่ากับที่สร้างใหม่ — ถ้าไม่ตรง ต้องอธิบายได้ว่าต่างเพราะอะไร (เช่น ระบบใหม่ไม่รวมคนไม่ฝาก ตามที่ตกลงไว้ใน Decision Log #6 ไม่ใช่บั๊ก)

👁️ **Pam ดู**: ตารางเทียบ "เก่า vs ใหม่" พร้อมคำอธิบายทุกจุดที่ต่างกัน

✅ **ผ่านเมื่อ**: ทุกจุดต่างกันอธิบายได้ตาม Decision Log ไม่มีจุดต่างที่อธิบายไม่ได้ (= บั๊ก)

---

## Stage 11 — Deploy จริง

🔨 **สร้าง**: ไม่มีโค้ดใหม่ — เตรียม Code.gs ฉบับสมบูรณ์ให้ Pam paste เข้า GAS editor (`AKfycbzScb5FNx-...` deployment ที่ใช้งานจริงตอนนี้ — ยืนยัน URL ตรงกับ `var API` ใน index.html ก่อนเสมอ) แล้ว Deploy → Manage deployments → Edit → version ใหม่ → Update จากนั้น commit+push index.html (frontend)

🧪 **ทดสอบ**: หลัง Pam paste+deploy แล้ว ให้ทดลองกดใช้งานจริงในเว็บอีกครั้ง (ซ้ำ Stage 8 บนเว็บจริงหลัง deploy)

👁️ **Pam ดู**: ผลการใช้งานจริงครั้งแรกบนเว็บ production

✅ **ผ่านเมื่อ**: ใช้งานได้จริงบนเว็บ production ไม่ใช่แค่ในสภาพแวดล้อมทดสอบ

---

## สิ่งที่ต้องการจาก Pam ระหว่างสร้าง

- Stage 2, 3, 6, 7, 10 เป็นจุดที่ต้องให้ Pam ช่วยยืนยันกับความจริงที่ Pam รู้เอง (ชื่อเด็ก, เลขบัญชี, ยอดเงินสดจริง) — ผมทำเองไม่ได้
- Stage 2 (การ match วันที่) และ Stage 10 (เทียบกับไฟล์จริง) คือจุดเสี่ยงสูงสุด จะขอเวลา/รายละเอียดเพิ่มตรงนี้

## จุดเสี่ยงที่ต้องระวังเป็นพิเศษ

1. **รูปแบบวันที่ไทย** ("D MMM YYYY" ไม่มี leading zero, เดือนต้องตรง `THAI_MONTHS` เป๊ะ) — ถ้า format ผิดแม้แต่นิดเดียว การกรองธุรกรรมจะได้ 0 แถวเงียบๆ โดยไม่มี error
2. **เลขบัญชีว่าง** — ต้องเว้นว่างได้โดยไม่ทำให้ทั้งแถวหาย
3. **สูตรใน Sheet2** ต้องอ้างอิงเซลล์ถูกตำแหน่งเป๊ะ (ถ้า Pam แทรก/ลบแถวทีหลัง สูตรอาจเพี้ยน — ควรใส่หมายเหตุเตือนไว้ในชีตว่า "ห้ามแทรกแถวในตารางนี้")
4. **ชื่อ sheet ซ้ำ** เมื่อสร้างทะเบียนวันเดียวกันซ้ำ — ของเดิมจะถูกลบทิ้งแบบไม่เตือน (ตรงตาม pattern เดิมของระบบ แต่ Pam ควรรู้ไว้)
